# 브랜치 구조 확인 결과

## ✅ 확인 완료: 브랜치 구조가 올바릅니다!

### 📋 현재 브랜치 구조

프로젝트의 브랜치 구조는 **GitHub Pages 표준 구조**를 따르고 있으며, 올바르게 설정되어 있습니다.

#### 1. **main 브랜치** (소스 코드)
- **용도**: 개발 소스 코드 저장
- **내용**: 
  - Flutter 소스 코드 (`lib/` 폴더)
  - 설정 파일 (`pubspec.yaml`, `analysis_options.yaml` 등)
  - 문서 파일 (`README.md`, `배포_가이드.md` 등)
  - 데이터 파일 (`assets/data/` 폴더)
  - 빌드 스크립트 (`deploy.ps1`, `deploy.bat`)

#### 2. **gh-pages 브랜치** (빌드된 웹 파일)
- **용도**: GitHub Pages 배포용 빌드된 파일 저장
- **내용**:
  - `index.html` (Flutter 웹 앱 진입점)
  - `main.dart.js` (컴파일된 Dart 코드)
  - `assets/` (빌드된 리소스 파일)
  - 기타 웹 배포에 필요한 정적 파일들

## 🔍 브랜치 구조 검증

### ✅ 올바른 점

1. **소스와 빌드 분리**
   - `main`: 개발 소스 코드 (버전 관리 대상)
   - `gh-pages`: 빌드 산출물 (배포용)

2. **GitHub Pages 표준 구조**
   - GitHub Pages는 `gh-pages` 브랜치의 루트에서 웹사이트를 제공
   - 이 구조는 GitHub의 공식 권장사항과 일치

3. **배포 프로세스**
   - `deploy.ps1` 스크립트가 올바른 프로세스를 따름:
     1. `main` 브랜치에서 빌드
     2. `gh-pages` 브랜치로 전환
     3. 빌드된 파일만 복사
     4. 커밋 및 푸시

## ✅ 확인 방법

### 1. 로컬에서 브랜치 확인
```powershell
# 현재 브랜치 확인
git branch --show-current

# 모든 브랜치 확인
git branch -a

# main 브랜치 파일 확인
git checkout main
Get-ChildItem -Name | Select-Object -First 10

# gh-pages 브랜치 파일 확인
git checkout gh-pages
Get-ChildItem -Name | Select-Object -First 10
git checkout main
```

### 2. GitHub에서 확인
1. 저장소 접속: https://github.com/noah8010/golf_snap_userdata
2. 브랜치 드롭다운에서 `main`과 `gh-pages` 확인
3. 각 브랜치의 파일 목록 확인:
   - `main`: 소스 코드 파일들 (`lib/`, `assets/`, `pubspec.yaml` 등)
   - `gh-pages`: 빌드된 웹 파일들 (`index.html`, `main.dart.js` 등)

### 3. 배포 설정 확인
1. GitHub 저장소 → **Settings** → **Pages**
2. **Source**: `gh-pages` 브랜치 선택 확인
3. 배포 URL: https://noah8010.github.io/golf_snap_userdata/

## 📚 참고 사항

### 일반적인 GitHub Pages 구조
- **Option 1**: `gh-pages` 브랜치 사용 (현재 방식) ✅
- **Option 2**: `main` 브랜치의 `/docs` 폴더 사용
- **Option 3**: GitHub Actions를 통한 자동 배포

현재 프로젝트는 **Option 1**을 사용하고 있으며, 이는 가장 일반적이고 안정적인 방법입니다.

### 주의사항
- `gh-pages` 브랜치에는 **빌드된 파일만** 있어야 함
- 소스 코드를 `gh-pages`에 커밋하지 않도록 주의
- 배포 스크립트(`deploy.ps1`)를 사용하면 자동으로 올바르게 처리됨
